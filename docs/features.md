# Features

## Authentication

**Google OAuth 2.0 (implicit flow)**
Users sign in with their Google account. The frontend handles the OAuth popup via `@react-oauth/google` and sends the resulting access token to the backend for validation. No passwords are stored. User profile picture and full name are pulled from Google and returned with the JWT response.

**JWT with auto-refresh**
Access tokens expire after 60 minutes. The Axios client automatically detects 401 responses and calls the refresh endpoint transparently. If the refresh token is also expired, the user is redirected to the login page.

## Chat

**Multi-session conversations**
Each user can have multiple independent chat sessions. Sessions are listed in the sidebar ordered by most recent activity. Sessions can be created and deleted at any time.

**Real-time streaming**
AI responses stream token-by-token via Server-Sent Events. The response appears incrementally as it is generated, with a blinking cursor during streaming and a bouncing dots indicator while waiting for the first token.

**Stop generation**
A stop button replaces the send button while streaming. Clicking it aborts the request via `AbortController`. The partial response received so far is kept in the UI.

**Context window**
The last 10 messages from the session are sent to the AI with every request, so the model can refer back to earlier parts of the conversation. The context window size is configurable in `settings.py`.

**Optimistic UI**
The user's message appears in the chat immediately when sent, before the server has confirmed receipt. It is replaced by the real server message when the `user_message` SSE event arrives. This eliminates any perceived latency on message send.

**Auto-title generation**
After the first message in a new session, a short title (4–6 words) is generated by the AI and applied to the session. The sidebar title updates live while the main response is still streaming.

**Regenerate response**
A regenerate button appears below the last assistant message. Clicking it deletes the last assistant message and re-runs the last user message through the same streaming flow, using whatever AI provider is available at that moment.

---

## AI Routing

**Multi-provider with automatic failover**
Three providers are supported: Groq (primary), Google Gemini (fallback), and Mistral (last resort). The router selects the first provider that has not exceeded its daily request limit. The switch between providers is invisible to the user.

**Provider and model metadata**
The provider name and model identifier are stored with every assistant message and displayed below each response in the UI (e.g. "groq · llama-3.3-70b-versatile").

---

## Markdown Rendering

Assistant messages are rendered as markdown with full support for:

- Headings, paragraphs, bold, italic
- Ordered and unordered lists
- Blockquotes and horizontal rules
- Inline code and fenced code blocks
- Tables
- Links (open in new tab)

**Code blocks** include syntax highlighting via `react-syntax-highlighter` using the One Dark theme, line numbers for blocks longer than 5 lines, and a copy button with a two-second checkmark confirmation.

## Progressive Web App (PWA)

The application is a fully compliant Progressive Web App. It can be installed on the user's device and continues to function when the network is unavailable.

**Install prompt**
On Android and desktop Chrome the browser's native install prompt is intercepted and surfaced as a polished slide-up banner at the bottom of the screen. The banner appears 3 seconds after the page loads to avoid disrupting the initial experience. Users can dismiss it; the dismissal is persisted in `sessionStorage` so it does not reappear in the same session.

On iOS Safari the browser does not fire `beforeinstallprompt`. Instead a contextual "Add to Home Screen" guide is shown — a modal that walks the user through the three-step share-sheet process. The guide is only shown once per session.

**Offline support**
When the device loses connectivity a slim red banner slides down from the top of the screen. When the connection is restored the banner briefly turns green ("Back online") before disappearing. Cached assets continue to serve the app shell; API calls that fail while offline surface the normal error states already present in the chat UI.

**Update flow**
When a new version of the service worker is available a floating toast appears in the top-centre of the screen with a "Reload" button. The update is never applied silently — the user always initiates it. The toast can also be dismissed; the update will be applied on the next natural page reload.

**App shortcuts**
Long-pressing the home screen icon (Android) or using the context menu (desktop) exposes two shortcuts defined in the manifest: "New Chat" (navigates to `/chat`) and "Dashboard" (navigates to `/dashboard`).

**First-install notice**
The first time the service worker finishes installing and the app is ready for offline use, a brief "App ready for offline use" toast is shown. It auto-dismisses after 4 seconds.

## Usage Dashboard

A dashboard at `/dashboard` shows platform-wide statistics for all users:

- Total messages, sessions, and registered users
- Provider breakdown — percentage of responses handled by each AI provider, shown as a bar chart
- Messages per day for the last 7 days, shown as a line chart
- Today's quota usage for each provider with a progress bar that turns red above 90%

The dashboard is accessible to all authenticated users.

---

## Rate Limiting

Each authenticated user is limited to 30 messages per hour by default (configurable via `RATE_LIMIT_PER_HOUR`). The limit applies to both the send and regenerate endpoints. When exceeded, the API returns a `429` response and the frontend displays the error message inline.

---

## Responsive Design

The layout is fully responsive. On desktop, the sidebar is static and always visible. On mobile, the sidebar is hidden by default and opens as a full-height drawer overlay via a hamburger menu. The design is mobile-first and tested on iOS Safari.

---

## Error Handling

- Stream errors are shown inline in the chat UI rather than as alerts or toasts
- An `ErrorBoundary` component wraps the entire app and catches any unhandled React render errors, showing a clean "Something went wrong" screen with a refresh button instead of a blank page
- The Axios client handles 401 errors globally with automatic token refresh
- AI provider failures fall back to the next provider silently
- Auto-title failures are swallowed silently — streaming continues unaffected
